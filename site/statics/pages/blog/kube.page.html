How to build a Kubernetes

Welcome to my series on how to build a "Kubernetes" (i.e. a container orchestration platform). The purpose of this series is to teach developers critical elements of writing distributed systems.

What is a container orchestrator?

A container orchestrator is a system that maintains an understanding of what containers the end-user would like running, and distributes the execution of those containers accross one or more computers.

The criteria container orchestrators use to decide on which computer it should execute a container varies, but typically consists of at least CPU and Memory constraints.

Why build a container orchestrator?

If you're here to learn how to build a better Kubernetes, you're probably here for the wrong reasons. Kubernetes is a solid, well supported, and nearly universally adopted platform. To compete with that your idea has to be leaps and bounds ahead.

There are two reasons I chose to build a container orchestrator for this series:

  1. I believe any distributed systems developer should have a cursory understanding of how they work.
  2. Container orchestrators model virtually every concept required to build a reliable distributed system.

What components will we build in this series?

The following components are the bare minimum to achieve what I would call a fully functional "container orchestrator":

  • Container Executive
  • API Server
  • CLI
  • State Storage
  • Overlay network
  • Ingress

Container Executive

A program that interacts with the container runtime (e.g. Docker) to download, launch, and maintain the state of containers expected to be running on an individual computer.

API Server

A program that receives commands sent by the end-user with the intention of establishing the desired state for the system.

CLI

A command line interface to simplify interaction with the API.

State Storage

A highly available system for storing the desired state of the container orchestration platform.

Overlay network

A virtual network for connecting containers based on routing rules specified by the end-user.

Ingress

A gateway between the network outside and the overlay network.
